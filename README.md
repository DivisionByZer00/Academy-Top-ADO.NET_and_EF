## Homework 01

### Тема: "Анатомия GUI приложения. Введение в Avalonia UI"

**Задание:** Вам необходимо спроектировать и реализовать структуру базы данных для небольшой библиотеки. Программа не требует написания C# кода; ваша задача — подготовить SQL-скрипт, который создает таблицы с правильными связями и наполняет их первичными данными.

**Техническое задание:** 

Таблица Genres (Жанры): Спроектируйте таблицу-справочник. Она должна хранить уникальный идентификатор (Id) и название жанра (Name). Название не должно быть пустым и должно быть уникальным.

Таблица Books (Книги): Спроектируйте таблицу для хранения книг. Каждая книга должна иметь:    
- Уникальный артикул/ID;
- Название (обязательно);
- Автор;
- Цену (число с плавающей точкой);
- Связь с жанром (GenreId).

Ограничения (Constraints):
- Установите внешний ключ (Foreign Key) между книгами и жанрами.
- Настройте правило: при удалении жанра, у связанных книг поле жанра должно становиться пустым (SET NULL).

**Совет по выполнению:**    
Вы можете использовать любой текстовый редактор (например, VS Code) для последовательного написания всех команд в один файл. После того как скрипт будет готов, вы можете файл целиком в SQL Editor в SQLiteStudio (Ctrl + O – запустить скрипты из файла) и запустить одной командой. Это гарантирует, что ваша база будет создана именно в той последовательности, которую вы задумали.

- ***Фильтрация групп:*** Выведите названия жанров, у которых средняя цена книг превышает 1000 единиц. Отсортируйте результат по убыванию цены.
- ***Пагинация:*** Напишите запрос, который имитирует получение данных для «второй страницы» каталога (вывести 3 книги, пропустив первые 3), отсортировав их по названию.

## Homework 05

### Тема: "Параметры, CRUD и транзакции"

**Задание:** Разработайте консольное приложение «Управление контактами с журналом действий». Приложение должно работать с SQLite через ADO.NET, выполнять добавление, изменение, удаление и поиск контактов, а также фиксировать изменения в отдельной таблице AuditLog. Главное условие: все изменяемые значения и фильтры передавайте только через параметры, а сценарий «изменение данных + запись в журнал» выполняйте в одной транзакции.

**Требования к проекту:**
- Тип проекта: Console App на .NET 9.0.
- Пакет для работы с SQLite: Microsoft.Data.Sqlite.
- Файл базы данных должен создаваться рядом с приложением через AppContext.BaseDirectory.
- Строка подключения должна храниться в одном месте: в константе, поле или свойстве класса Program.
- В проекте должна быть модель Contact с полями Id, FullName, Phone, Email, Department.
- Допустимо добавить модель AuditLogEntry, если так удобнее читать журнал в объекты и выводить его в консоль.
- Проект должен запускаться повторно без ошибок и без бесконечного дублирования тестовых записей.
- Для соединения, команд, reader и транзакций используйте using var там, где это уместно.
- Запрещено собирать SQL-команды конкатенацией строк для данных, которые меняются во время работы программы.

**Структура данных:**
- Создайте таблицу Contacts, если её ещё нет.
- Минимальный набор полей таблицы Contacts: Id, FullName, Phone, Email, Department.
- Id — целочисленный первичный ключ с автоинкрементом. FullName — обязательное поле. Остальные поля можно оставить допускающими NULL.
- Создайте таблицу AuditLog, если её ещё нет.
- Минимальный набор полей таблицы AuditLog: Id, ContactId, ActionName, Details, CreatedAt.
- ContactId должен хранить идентификатор записи, с которой связано действие. ActionName — короткое имя операции, например CREATE, UPDATE или DELETE. Details — текст с пояснением. CreatedAt — дата и время в строковом виде.
- Если таблица Contacts пуста, программа должна добавить не менее 8 тестовых контактов. В тестовых данных должно быть не менее 3 разных отделов.
- Для проверки обработки NULL обязательно добавьте хотя бы один контакт без телефона, хотя бы один без e-mail и хотя бы один без отдела.

**Что должна уметь программа:**
- Выводить полный путь к файлу базы данных.
- Выводить исходный список всех контактов.
- Добавлять новый контакт через отдельный метод с параметрами SQL-команды.
- Изменять существующий контакт по Id через отдельный метод с параметрами SQL-команды.
- Удалять контакт по Id через отдельный метод с параметрами SQL-команды.
- Выполнять поиск контактов по строке. Поиск должен работать минимум по двум полям, например FullName и Department. Для LIKE тоже используйте параметр, а не подстановку текста в SQL.
- Показывать результат каждой операции в консоли: сколько записей затронуто, какой контакт добавлен, что изменилось после UPDATE, исчезла ли запись после DELETE.
- Выводить содержимое таблицы AuditLog после успешных операций изменения данных.
- Продемонстрировать успешную транзакцию. Пример: создание контакта и запись строки в AuditLog должны выполниться вместе и завершиться commit.
- Продемонстрировать rollback. Пример: после вставки контакта искусственно вызовите ошибку во второй команде или выбросьте исключение до commit. После этого в базе не должно остаться ни нового контакта, ни записи в журнале.
- После всех сценариев программа должна вывести итоговый список контактов и итоговый журнал действий.

**Рекомендуемая структура кода:**
- EnsureDatabase() — создаёт таблицы и при первом запуске добавляет тестовые записи.
- GetAllContacts() — возвращает список всех контактов.
- SearchContacts(string searchText) — выполняет поиск по нескольким полям через параметры.
- AddContact(Contact contact) — добавляет запись и возвращает количество затронутых строк или новый Id.
- UpdateContact(Contact contact) — изменяет запись по Id.
- DeleteContact(int id) — удаляет запись по Id.
- GetAuditLog() — читает журнал действий и возвращает список записей или выводит их в консоль.
- RunSuccessfulTransaction(...) и RunRollbackTransaction(...) — отдельные методы для демонстрации commit и rollback. Допустимо выбрать другие имена, но логика должна быть разделена и понятна.
- PrintContacts(...) и PrintAuditLog(...) — отдельные методы для аккуратного консольного вывода.

**Технические требования:**
- Для INSERT, UPDATE и DELETE используйте ExecuteNonQuery().
- Для чтения списков контактов и журнала используйте ExecuteReader() и цикл while (reader.Read()).
- Если нужно проверить количество строк перед заполнением базы, используйте ExecuteScalar().
- Все переменные значения передавайте через параметры: имя, телефон, e-mail, отдел, Id, поисковую строку и текст журнала.
- В запросах UPDATE и DELETE обязательно проверяйте количество затронутых строк. Если строка не найдена, программа должна вывести понятное сообщение.
- Для транзакции используйте BeginTransaction(). Команды, которые входят в одну операцию, должны работать через одно и то же соединение и одну и ту же транзакцию.
- При успешном выполнении всех команд вызывайте Commit(). При ошибке — Rollback(). Обработку ошибки покажите через try/catch.
- При чтении nullable-полей таблицы Contacts проверяйте reader.IsDBNull(...). Не считывайте NULL как обычную строку без проверки.
- Код должен быть разбит на небольшие методы. Не размещайте всюлогику работы с базой внутри одного длинного Main().
- Меню с бесконечным циклом не требуется. Достаточно последовательно вызвать методы и показать результат их работы.

## Homework 06

### Тема: "DataAdapter, DataTable и CommandBuilder"

**Задание:** Разработайте консольное приложение «Редактирование справочника отделов через DataTable». Приложение должно работать с SQLite через ADO.NET, загружать таблицу Departments в DataTable, изменять строки в памяти и сохранять изменения обратно в базу данных через SQLiteDataAdapter и SQLiteCommandBuilder. Главное условие: до сохранения нужно показать состояния строк Added, Modified и Deleted, а после Update() повторно загрузить таблицу из базы и вывести итоговый результат.    
Это задание закрепляет три идеи темы. Первая — отсоединённая модель: приложение редактирует локальную таблицу в памяти, а не поток данных из DataReader. Вторая — DataRow хранит не только значения, но и состояние изменения. Третья — при работе с одной простой таблицей команды INSERT, UPDATE и DELETE можно получить автоматически через CommandBuilder.

**Требования к проекту:**
- Тип проекта: Console App на .NET 9.0.
•Пакет для работы с SQLite: System.Data.SQLite или System.Data.SQLite.Core.
- Пространства имён в коде: System.Data и System.Data.SQLite.
- Файл базы данных должен создаваться рядом с приложением через AppContext.BaseDirectory.
- Строка подключения должна храниться в одном месте: в константе, поле или свойстве класса Program.
- Проект должен запускаться повторно без ошибок и без бесконечного дублирования стартовых данных.
- Для соединения, адаптера, команд и других IDisposable-объектов используйте using var там, где это уместно.
- Команды сохранения не нужно писать вручную: InsertCommand, UpdateCommand и DeleteCommand должны формироваться через SQLiteCommandBuilder.
- Запрещено подменять тему задания ручными SQL-командами для обновления строк. Основной сценарий должен проходить через DataTable + DataAdapter + CommandBuilder.

**Структура данных:**
- Создайте таблицу Departments, если её ещё нет.
- Минимальный набор полей таблицы Departments: Id, Name.
- Id — целочисленный первичный ключ с автоинкрементом.
- Name — обязательное текстовое поле. Допустимо добавить ограничение UNIQUE, если хотите запретить дубли отделов.
- Если таблица Departments пуста, программа должна добавить не менее 5 тестовых отделов.
- Стартовые данные должны быть достаточны для демонстрации трёх действий: изменение существующей строки, добавление новой строки и удаление существующей строки.

**Что должна уметь программа:**
- Выводить полный путь к файлу базы данных.
- Создавать таблицу Departments и заполнять её начальными данными при первом запуске.
- Создавать SQLiteDataAdapter с запросом SELECT Id, Name FROM Departments ORDER BY Id.
- Загружать данные в DataTable через метод Fill().
- Настраивать адаптер так, чтобы таблица в памяти знала о первичном ключе.

Для этого используйте MissingSchemaAction = AddWithKey или вручную настройте PrimaryKey после загрузки.

- Выводить исходное содержимое таблицы после Fill().
- Показывать для каждой строки не только значения, но и RowState.
- Изменять имя одного существующего отдела по Id.
- Добавлять новый отдел в DataTable через NewRow() и Rows.Add().
- Помечать один отдел на удаление через Delete().
- До вызова Update() выводить локальное состояние таблицы так, чтобы было видно минимум одну строку со статусом Modified, одну со статусом Added и одну со статусом Deleted.
- После подготовки изменений создавать SQLiteCommandBuilder для того же адаптера и вызывать Update(table).
- Показывать количество записей, которые были обработаны при сохранении.
- После сохранения заново читать таблицу из базы в новый DataTable и выводить итоговое содержимое.
- Подтверждать, что после повторной загрузки строки имеют состояние Unchanged.
- В конце программы вывести 2–3 предложения с кратким выводом: для каких таблиц подход DataTable + CommandBuilder подходит, а где для проекта удобнее переходить к EF Core.

**Рекомендуемая структура кода:**
- EnsureDatabase() — создаёт таблицу Departments и добавляет стартовые записи при первом запуске.
- CreateAdapter(SQLiteConnection connection) — создаёт и настраивает SQLiteDataAdapter.
- LoadDepartments(SQLiteDataAdapter adapter) — загружает DataTable и возвращает её вызывающему коду.
- ShowTable(DataTable table, string title) — выводит данные и состояния строк в консоль.
- RenameDepartment(DataTable table, int id, string newName) — изменяет строку по ключу.
- AddDepartment(DataTable table, string name) — добавляет новую строку.
- DeleteDepartment(DataTable table, int id) — помечает строку на удаление.
- SaveChanges(SQLiteDataAdapter adapter, DataTable table) — создаёт CommandBuilder и вызывает Update().
- ReloadDepartments() — повторно считывает таблицу из базы для проверки результата.
- PrintConclusion() — выводит краткий вывод по применимости отсоединённой модели.

**Технические требования:**
- DataTable должен использоваться как основная структура для редактирования данных в памяти.
- Для поиска строки по Id удобно использовать Rows.Find(id), но этот вариант требует корректной информации о ключе в DataTable.
- Если вы выводите строку со статусом Deleted, используйте версию Original. Текущие значения для такой строки уже недоступны.
- Не вызывайте AcceptChanges() вручную до Update(). Иначе таблица потеряет сведения о том, какие строки были добавлены, изменены или удалены.
- SelectCommand адаптера должен работать с одной физической таблицей и включать поле ключа Id. Иначе CommandBuilder не сможет корректно сгенерировать команды сохранения.
- Соединение должно открываться корректно и освобождаться после завершения операций.
- Основная логика не должна быть размещена целиком в одном длинном Main(). Разбейте решение на небольшие методы.
- Меню с бесконечным циклом не требуется. Достаточно последовательно выполнить сценарий: загрузка, локальные изменения, сохранение, повторная загрузка, вывод результата.
- Обработку ошибок покажите через try/catch. При ошибке программа должна выводить понятное сообщение, а не завершаться без объяснения.
- Допустимо вывести в консоль SQL-команды, которые генерирует CommandBuilder, если вы хотите дополнительно проверить результат, но это не является обязательным требованием.
